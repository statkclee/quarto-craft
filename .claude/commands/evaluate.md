# /evaluate — 생성된 문서 품질 평가

## 개요
docs/ 폴더의 렌더링 결과물을 브라우저에서 열고, 시각적·구조적 품질을 평가한다.

## 평가 절차

### 1. 렌더링 결과물 확인
docs/ 하위의 모든 HTML/PDF 파일을 브라우저에서 열어 확인한다.

### 2. 평가 항목 (5개 영역, 각 10점 만점)

#### A. 타이포그래피 (10점)
- 폰트 로딩 여부 (Pretendard, Noto Serif KR, JetBrains Mono / D2Coding)
- 제목 계층 구분 (H1~H6 크기·굵기·간격 차이)
- 본문 가독성 (행간, 자간, 문단 간격)
- 한국어 줄바꿈 (word-break: keep-all 적용 여부)
- 코드 폰트 구분 (인라인/블록)

#### B. 색상 & 시각 디자인 (10점)
- Primary 색상 일관성
- 콜아웃 변형별 색상 구분
- 배경/텍스트 대비 (WCAG AA 기준)
- 링크 색상 및 호버 상태
- 코드 블록 배경/보더 처리

#### C. 레이아웃 & 구조 (10점)
- TOC 위치 및 스타일
- 제목 번호 매기기
- 표/그림 캡션 위치
- 마진/패딩 일관성
- 반응형 대응 (좁은 화면)

#### D. 컴포넌트 품질 (10점)
- 콜아웃 (note/tip/warning/important/caution)
- 표 (헤더, 줄무늬, 호버)
- 코드 블록 (구문 강조, 스크롤)
- 인용문 (서체 변경, 보더)
- 각주/참조

#### E. 템플릿별 완성도 (10점)
- report: 보고서다운 구성 (abstract, 목차, 섹션)
- presentation: 슬라이드 전환, 레이아웃
- dashboard: valuebox, 차트 배치
- book: 챕터 네비게이션
- typst-pdf: 인쇄 품질, 여백, 폰트

### 3. 평가 결과 정리
각 영역별 점수와 구체적 문제점, 개선 방안을 기록한다.

### 4. 기술문서 작성
`tech_document/YYYY-MM-DD_문서품질평가-v{N}.md` 형식으로 저장한다.

### 5. 개선 작업
평가 결과를 바탕으로 디자인 토큰, 컴포넌트, 템플릿을 수정한다.
수정 후 다시 렌더링하여 개선 여부를 확인한다.
